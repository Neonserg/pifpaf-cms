# Робота з кодом

Кілька виконавців (люди і/або ШІ-агенти) можуть працювати над репозиторієм одночасно. Ці правила існують, щоб зміни не конфліктували і залишались зрозумілими.

## Гілки і Pull Request

- `main` захищена: прямі коміти заборонені, злиття лише через Pull Request.
- Одна задача — одна feature-гілка (`feature/назва`, `fix/назва`).
- Перед злиттям PR повинен пройти CI (лінт, перевірка типів, збірка) і рев'ю.

## Рев'ю

Кожен PR перевіряється (людиною і/або іншою ШІ-моделлю) перед злиттям. Критерій прийняття — проходження CI plus логічна відповідність задачі, під яку зроблено зміну.

## Стиль коду

- TypeScript, ESLint-конфігурація з `eslint-config-next` — виправляти попередження до злиття, не вимикати правила без причини.
- Дизайн-токени (`styles/tokens.css`) — єдине джерело кольорів/відступів; не хардкодити значення напряму в компонентах.

## Зміни схеми БД

- Будь-яка зміна структури БД — окрема іменована міграція через Supabase MCP `apply_migration`, ніколи вручну через дашборд.
- Після кожної міграції перегенерувати `lib/supabase/database.types.ts` (`generate_typescript_types`) і закомітити разом зі змінами.
- Перевіряти `get_advisors` (security) після кожної зміни схеми — нові попередження або виправляти, або явно документувати чому вони навмисні (як `form_submissions.INSERT`).

## Коміти

Короткий опис у наказовому способі ("Add gallery layout switch", "Fix slug editing on home page"), без загальних формулювань на кшталт "update files".

## Версіонування ядра

Якщо репозиторій використовується як основа для нової незалежної інсталяції іншого сайту (див. ARCHITECTURE.md, розділ "Мультиінсталяційність") — фіксувати релізи тегами і вести `CHANGELOG.md`, щоб було зрозуміло, які оновлення можна перенести на вже розгорнуті інсталяції.
